Dear Dan,

if you're still in the design-phase of Etm-new, here is something I
would like to share...

Some time ago I suggested to add Taskwarrior-like feauture for handling
priorities, but probably I did not explain the feature very well and
you responded you didn't get it.

Now, few days ago, I had kind of "Aha moment" which possibly explain
the main difference I see in e.g. Etm and similar apps like Emacs'
org-mode vs Taskwarrior and some of its clones...

It just just that priority feature!

For instance - in the case of e.g. John, it is obvious that he needs
tool which can remind him of his strict daily agenda showing court
hearing, client appointments etc. I'd call that kind of agenda "static"
in the sense that it is driven by user's specification of
tasks/reminders. And such setup is perfectly valid for such users.


On the other hand, let's imagine freelance worker who has several
clients and has to decide for which client he/she should do the work.

In such cases, Taskwarrior's ability to sort tasks by
dynamically-building priorities is indispensable.

If you can take look at this 7min video

https://www.youtube.com/watch?v=Y5nWB9OSfzUhttps://www.youtube.com/watch?v=Y5nWB9OSfzU

I believe it will become clear how such feature - let's call it
"dynamic agenda" can be useful or can be used.

For the explanation how it can be achieved I'll copy short description
from the Tasklite - one the Taskwarrior's clones
(https://tasklite.org/concepts/main.html#priority) where it is stated:

"The priority of a task is a decimal number between negative and
positive inifity. It is automatically calculated based on the age, the
due date, and several other values.

Priority

The idea is that you never have to manually set a priority, because it
can be derived accurately from other values. This of course requires
you to use the other available meta information adequately!

The exact calculation algorithm can be found in the SQL view tasks_view
in the [Migrations.hs] file.

If you want to adjust the priority of selected tasks manually, you can
use the tl boost [ulid] command to increase the priority by 1, or the
tl hush [ulid] command to decrease it by 1."

So, the question is what do you think whether such mechanism could be
added as optional feature to Etm-new's design in a sense that those
user who do not need it, do not have to use it?


Sincerely,
Gour

"Taskwarrior uses a polynomial expression to calculate urgency, which is
influenced by various factors including priority, due dates, and waiting
status. The priority of a task is a significant factor in urgency
calculations, with higher priorities contributing more to the urgency
score. For instance, a high priority task (H) has a coefficient of 6.0,
medium priority (M) has 3.9, and low priority (L) has 1.8. Even tasks
with no assigned priority contribute to urgency, indicating that no
priority is considered more urgent than a low priority.

Coefficients for these factors can be adjusted to better fit individual
preferences. For example, if you consider priority to be the most
important factor, you might increase the value from 6.0 to 10.0 for
high priority tasks. Adjustments should be made carefully to avoid one
term dominating the urgency calculation.

Taskwarrior allows users to define their own priority values and their
order of importance through the uda.priority.values configuration. For
instance, setting uda.priority.values=H,M,,L configures the priorities
such that no priority is higher than a low priority.

The urgency score is used to sort tasks in the next report, which lists
tasks by decreasing urgency. This helps users identify which tasks
they should work on next based on their urgency scores."

Here's a first pass.
 
For tasks, the status would be either "scheduled", "next", or "waiting". Scheduled tasks would have a due_date such as "20250331". Next tasks correspond to unscheduled tasks to be done as soon as time permits. Waiting tasks would be reviewed periodically. Here's the important part of the SQLite Table setup:
 
CREATE TABLE tasks (
    id INTEGER PRIMARY KEY,
    description TEXT NOT NULL,
    status TEXT NOT NULL CHECK (status IN ('scheduled', 'next', 'waiting')),
    due_date TEXT,  -- Format: 'YYYYMMDD', required for scheduled
    priority REAL NOT NULL CHECK (priority BETWEEN 0.0 AND 10.0)
);
 
Priority would actually be computed  from Account, Project and Task priorities but showing that here would only complicate things unnecessarily. 
 
Here's an example query for a "now view" that gathers tasks that have
  • status "scheduled" and  a due_date < tomorrow (past due or due today)
  • or status "next"
and then sorts them
  • first by status: "scheduled" then "next"
  • then by due date and then priority for "scheduled"
  • finally by priority for "next"
SELECT *
FROM tasks
WHERE
    (status = 'scheduled' AND due_date < '20250401')
    OR status = 'next'
ORDER BY
    CASE status
        WHEN 'scheduled' THEN 0
        WHEN 'next' THEN 1
    END,
    due_date,
    priority DESC;

Here’ an example of the sorting process:
---
Table Structure:
CREATE TABLE tasks (
    id INTEGER PRIMARY KEY,
    description TEXT NOT NULL,
    status TEXT CHECK (status IN ('scheduled', 'next', 'waiting')),
    due_date TEXT,  -- Format: 'YYYYMMDD'
    priority REAL CHECK (priority BETWEEN 0.0 AND 10.0)
);
---
Sample Data (Assuming Today = 2025-04-02):
| id | description        | status     | due_date  | priority |
|----|--------------------|------------|-----------|----------|
| 1  | Do taxes           | scheduled  | 20250401  | 9.5      |

| 3  | Pay rent           | scheduled  | 20250403  | 10.0     |
| 4  | Backup computer    | scheduled  | 20250330  | 5.0      |
| 5  | Fix bug            | next       | NULL      | 8.0      |
| 6  | Read proposal      | next       | NULL      | 6.5      |
| 7  | Update resume      | scheduled  | 20250402  | 6.0      |
---
Sorting Logic:
- Show scheduled tasks with due_date < '20250403', ordered by due_date and priority DESC
- Then show tasks with status "next", ordered by priority DESC
---
Query:
SELECT *
FROM tasks
WHERE
    (status = 'scheduled' AND due_date < '20250403')
    OR status = 'next'
ORDER BY
    CASE status
        WHEN 'scheduled' THEN 0
        WHEN 'next' THEN 1
    END,
    due_date,
    priority DESC;

In TaskWarrior the most important urgency components are (1) having a "next" tag which gets an urgency component of 15 and (2) having a due date which gets a maximum urgency of 12. The intent seems to be to have the "next" tasks always at the top of the default (next) list with other pending tasks sorted by their urgency. This places unfinished tasks with due dates falling on or before the current date near the top of the default "next" list.  
Illustrative coefficients and urgency values are as follows:
 
```python
  urgency.user.tag.next.coefficient           15.0 # +next tag
  urgency.due.coefficient                        12.0 # overdue or near due date
  urgency.blocking.coefficient                   8.0 # blocking other tasks
  urgency.uda.priority.H.coefficient           6.0 # high Priority
  urgency.uda.priority.M.coefficient           3.9 # medium Priority
  urgency.uda.priority.L.coefficient            1.8 # low Priority
  urgency.scheduled.coefficient                 5.0 # scheduled tasks
  urgency.active.coefficient                       4.0 # already started tasks
  urgency.age.coefficient                          2.0 # coefficient for age
  urgency.annotations.coefficient              1.0 # has annotations
  urgency.tags.coefficient                         1.0 # has tags
  urgency.project.coefficient                     1.0 # assigned to any project
  urgency.user.project.My Project.coefficient  5.0 # assigned to project:"My Project"
  urgency.waiting.coefficient                    -3.0 # waiting task
  urgency.blocked.coefficient                   -5.0 # blocked by other tasks`
```
 
For tasks with a due date, the contribution of due to the urgency of the task is calculated as follows:
 
```python
def urgency_due(due: datetime) -> float:
    """
    This function calculates the urgency coefficient for a task based on its due datetime relative 
    to the current datetime and returns a float value between 0.2 (due >= now + 14 days) and 1.0 
    (due <= now - 7 days). This coefficient is then multiplied by the urgency.due.coefficient (12.0)
    to get the due contribution to the overall urgency of the task.
    """
    if not due or not isinstance(due, datetime):
        return 0.0
    now = datetime.now()
    days_overdue = (now - due).total_seconds() / 86400.0
    if days_overdue >= 7.0:
        return 1.0  # < 1 wk ago
    elif days_overdue >= -14.0:
        return ((days_overdue + 14.0) * 0.8 / 21.0) + 0.2
    else:
        return 0.2  # > 2 wks
```
 
Note that the 14 days, the 7 days and the 0.2 - 1.0 range are hard coded in TaskWarrior - the only user configuration variable is the urgency.due.coefficient (12.0). Here is the range of values when due differs from now by an integer number of days between -7 and +14:
